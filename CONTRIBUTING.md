## 🌿 Гілки (Branches)

### Основні гілки

| Гілка | Призначення |
|-------|-------------|
| `main` | Стабільна продакшн-версія. Прямі пуші **заборонені**. |
| `develop` | Інтеграційна гілка. Всі PR зливаються сюди. |

### Робочі гілки

Кожна задача виконується у власній гілці. Назва гілки **завжди** складається з префіксу і короткого опису:

```
<prefix>/<короткий-опис-через-дефіс>
```

#### Префікси

| Префікс | Коли використовувати | Приклад |
|---------|----------------------|---------|
| `feature/` | Новий функціонал | `feature/grade-journal-ui` |
| `fix/` | Виправлення помилки | `fix/login-redirect-loop` |
| `chore/` | Налаштування, залежності, конфіги | `chore/eslint-setup` |
| `refactor/` | Рефакторинг без зміни поведінки | `refactor/auth-middleware` |
| `docs/` | Тільки документація | `docs/api-endpoints` |

#### Правила іменування

- ✅ Тільки **малі літери** та дефіси
- ✅ Коротко, але зрозуміло (3–5 слів)
- ✅ Описує **що робиться**, а не хто робить
- ❌ `marian-fixes-stuff`, `new-branch`, `test123`

```bash
# ✅ Правильно
git checkout -b feature/redcoins-transaction-history
git checkout -b fix/student-grade-display
git checkout -b chore/add-dotenv-validation

# ❌ Неправильно
git checkout -b MyFeature
git checkout -b fix
git checkout -b marian/task
```

## ✍️ Commit Messages

Ми використовуємо [Conventional Commits](https://www.conventionalcommits.org/).

### Формат

```
<type>(<scope>): <короткий опис>

[необов'язковий детальний опис]

[необов'язкова мітка: Closes #номер]
```

### Типи (type)

| Type | Коли використовувати |
|------|----------------------|
| `feat` | Новий функціонал |
| `fix` | Виправлення бага |
| `chore` | Рутинні задачі (оновлення залежностей, конфіги) |
| `refactor` | Рефакторинг без зміни поведінки |
| `docs` | Зміни в документації |
| `style` | Форматування, відступи (без зміни логіки) |
| `test` | Додавання або виправлення тестів |

### Область (scope)

Вказує на частину проекту: `frontend`, `backend`, `db`, `auth`, `grades`, `redcoins`, `shared` тощо.

### Приклади

```bash
# ✅ Правильно
feat(backend): add POST /grades/bulk endpoint
fix(frontend): correct grade display for absent students
chore(deps): update mongoose to 8.x
docs(api): add swagger annotations to auth routes
refactor(auth): extract token validation to middleware
style(frontend): fix indentation in GradeJournal component

# ✅ З тілом та закриттям issue
feat(redcoins): add transaction history endpoint

Реалізовано GET /redcoins/transactions з пагінацією.
Повертає список транзакцій для поточного студента.

Closes #42

# ❌ Неправильно — занадто розпливчато
git commit -m "fix stuff"
git commit -m "update"
git commit -m "changes"
git commit -m "WIP"
```

### Правила написання

- ✅ **Перший рядок** — до 72 символів, imperative mood ("add", "fix", не "added", "fixed")
- ✅ **Тільки одна зміна** на коміт — не змішуй feature + refactor
- ✅ Якщо коміт закриває задачу — додай `Closes #<номер>`
- ❌ Не роби масивних комітів з сотнями рядків

## 🔄 Workflow: від задачі до злиття

```
Issue → Branch → Commits → Pull Request → Code Review → Merge
```

### Покроково

1. **Обери задачу** на GitHub Project Board і переведи її в `In Progress`
2. **Створи гілку** від `develop`:
   ```bash
   git checkout develop
   git pull origin develop
   git checkout -b feature/your-feature-name
   ```
3. **Роби невеликі атомарні коміти** в процесі роботи
4. **Перед PR** — self-review: перечитай свій diff, запусти лінтер:
   ```bash
   npm run lint        # перевір помилки
   npm run dev         # переконайся, що все запускається
   ```
5. **Відкрий Pull Request** на `develop`, заповни шаблон
6. **Дочекайся code review** від ментора (`@Vybyranyi`)
7. **Виправ зауваження** (якщо є) новими комітами
8. **Після approval** — ментор зливає PR

## 👀 Code Review

### Чи потрібен?

**Так, завжди.** Кожен PR повинен бути схвалений ментором (`@Vybyranyi`) перед злиттям. Це налаштовано автоматично через [CODEOWNERS](.github/CODEOWNERS).

> **Чому?** Code review — це не контроль, а навчання. Це один з найважливіших навичок, якому вчить цей проект.

### Що перевіряється

| Категорія | Що дивимось |
|-----------|-------------|
| **Функціональність** | Чи виконує код те, що описано в задачі |
| **Читабельність** | Чи легко зрозуміти код без зайвих пояснень |
| **Іменування** | Змінні, функції, компоненти названі зрозуміло |
| **Структура** | Файли лежать у правильних місцях |
| **Стиль** | Відповідність ESLint/Prettier правилам |
| **Безпека** | Немає хардкод-токенів, секретів у коді |

### Для студента (автора PR)

- Заповни PR-шаблон повністю
- Зроби **self-review** перед тим як запросити ревʼю
- Відповідай на коментарі — погоджуйся або аргументуй
- Не роби force push після того як ревʼю розпочато
- Виправлення роби **новими комітами**, не переписуй історію

### Для ментора (рев'юера)

- Надавай **конструктивні** коментарі з поясненням "чому"
- Розрізняй `blocker` (треба виправити) і `nit` (побажання)
- Approved PR зливає ментор через **Squash and Merge** або **Merge Commit**

## 🚀 Локальний запуск

```bash
# Встановити залежності
npm install

# Запустити всі частини (shared + backend + frontend)
npm run dev
```

> Перед запуском переконайся, що `.env` файли налаштовані. Приклади значень шукай у `.env.example` або запитай у ментора.

## 🎨 Стиль коду

- **Форматер**: Prettier — запускається автоматично або вручну
- **Лінтер**: ESLint — перевір помилки перед PR
- **Мова коду**: англійська (назви змінних, функцій, коментарі в коді)
- **Мова комунікації**: українська (PR описи, коментарі в review, коміт повідомлення)
